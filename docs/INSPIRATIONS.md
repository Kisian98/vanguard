# Inspirations

This document lists projects, systems, and ideas that may influence Vanguard.

The goal is to learn from adjacent work without copying names, interfaces, code, branding, wording, or structure. Inspiration should be visible in the thinking, not in direct imitation.

## Principle

Vanguard can study other projects for:

- useful patterns,
- technical direction,
- interface ideas,
- offline-first design,
- field-computing assumptions,
- documentation structure,
- and operational mood.

Vanguard should not copy:

- project names,
- logos,
- visual identity,
- exact layouts,
- fictional organizations,
- code,
- documentation text,
- or product claims.

If an idea is borrowed, it should be transformed until it clearly serves Vanguard's own purpose.

## Nearby Projects and Systems

### SolarSentinel

SolarSentinel is close in spirit because it combines field networking, offline capability, solar-powered deployment, radio concepts, mesh communication, and MilSim or training-adjacent use cases.

Vanguard can learn from:

- field-node thinking,
- degraded-network assumptions,
- lightweight distributed systems,
- tactical information flow,
- and practical off-grid constraints.

Vanguard should remain distinct by focusing on the operator workstation experience, product identity, desktop environment, and future reproducible OS implementation.

Reference: https://github.com/CptNope/Solar-Sentinel

### CyberDeck

CyberDeck is relevant as an offline workstation and portable knowledge system. It focuses on local resources, local AI, maps, survival information, and network-light operation.

Vanguard can learn from:

- offline-first design,
- local knowledge access,
- portable workstation assumptions,
- and useful field tools that do not depend on cloud services.

Vanguard should remain distinct by treating the workstation as a coherent fictional product platform, not only a collection of useful utilities.

Reference: https://github.com/sarogamedev/CyberDeck

### OSINT War Room, World Monitor, and SitDeck-style dashboards

These projects are useful references for situation-room interfaces, map layers, public feeds, dashboards, and information-dense visual presentation.

Vanguard can learn from:

- dashboard composition,
- feed aggregation,
- map-centered interfaces,
- alert panels,
- and data layering.

Vanguard should avoid becoming a surveillance, intelligence, or real command-and-control platform. Public data should be used carefully, and fictional framing should remain clear.

Reference: https://github.com/Hue-Jhan/OSINT-War-Room

### TAK, ATAK, WinTAK, iTAK, and FreeTAKServer

TAK systems are serious real-world situational-awareness tools. They are useful to study because they show how field operators think about maps, teams, markers, chat, location, and shared operational pictures.

Vanguard can learn from:

- geospatial workflows,
- operator-centered map interaction,
- team status concepts,
- marker systems,
- and field communication patterns.

Vanguard should remain clearly separate from real operational command systems. It should be a fictional MilSim, demonstration, and technical experimentation platform, not a tool for real-world targeting, surveillance, or tactical coordination.

Reference: https://www.civtak.org/

### NixOS and Hyprland workstation configurations

Many public projects already combine NixOS, flakes, Hyprland, Wayland tools, custom panels, launchers, and desktop theming.

Vanguard can learn from:

- reproducible system configuration,
- clean module structure,
- desktop composition,
- keyboard-driven workflows,
- and maintainable dotfile organization.

Vanguard should remain distinct by turning this stack into a coherent field workstation concept instead of another generic Linux rice.

Reference example: https://github.com/JaKooLit/NixOS-Hyprland

## Working Direction

The useful space for Vanguard is not simply "tactical UI".

Vanguard should aim to become a fictional but usable field workstation platform with:

- a clear product identity,
- a reproducible technical foundation,
- offline usefulness,
- restrained atmosphere,
- operator-focused workflows,
- event-ready presentation,
- and honest boundaries around what the project is not.

The standard is simple: learn broadly, build originally.
